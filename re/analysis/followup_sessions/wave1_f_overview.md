## Task 1 — Signature refinement (`wave1_f_signature_refinement_worklist.csv`)

Columns:
- `orig_address` — Ghidra virtual addr (8-hex, lowercase, `0x` prefix)
- `port_function_name` — port symbol name
- `port_signature` — `<ret> name(<params>)` as it appears in port source
- `signature_quality_hint` — `good` / `partial` / `weak`

Quality breakdown:
- 148 `good` — fully-typed params (e.g. `TD5_Actor *actor`)
- 32 `partial` — mixed typed + generic primitives
- 126 `weak` — mostly `int` / `void *` (port hasn't refined yet)

Wave 2 consumption: iterate `good`/`partial` rows first and call
`mcp__ghidra__function_signature_set` (with `parameter_replace` /
`parameter_add` for individual param renames if signature already partial).
For `weak` rows, skip unless caller knows the right semantic types.

Selection heuristic when multiple port functions reference the same orig
addr: prefer non-pilot/non-trace functions, then higher quality, then
larger body. Pilot/trace stubs are correctly filtered out (0 pilot/trace
functions in output despite 60+ trace files in the source tree).

## Task 2 — Comment-sync plan (`wave1_f_comment_sync_plan.csv`)

Columns:
- `orig_address` — Ghidra virtual addr (may be blank for ARCH-DIVERGENCE
  markers with no nearby address — 75 rows / 10.7%)
- `comment_kind` — `CONFIRMED` (325) or `ARCH-DIVERGENCE` (378)
- `port_file`, `port_line` — provenance
- `comment_text_short` — one-line ~150-char summary

Wave 2 consumption: for each row with non-blank `orig_address`, call
`mcp__ghidra__comment_set` with `kind=PRE_COMMENT` (or `PLATE_COMMENT`
for function-header attachments) pointing at the orig address. Multiple
rows for the same address are valid — append them as a multi-line block
rather than overwriting.

The 75 rows without addrs are commentary/manifest references; skip them
for Wave 2 (no actionable Ghidra target). Most originate from
`main.c:895-925` (CRT/LIBC bridge manifest preamble).

## Task 3 — Local-variable naming targets (`wave1_f_local_naming_targets.csv`)

Columns:
- `orig_address` — Ghidra virtual addr of the orig function
- `port_file`, `port_line` — port-side location of the local decl
- `suggested_name` — port's chosen variable name (likely informative)
- `suggested_type` — port's type as written
- `context_snippet` — the port source line (1-line, max 120 chars)

42 unique orig functions covered. 25/42 funcs have the full 6 locals
captured; the rest have fewer because the port body has fewer named
declarations (e.g. inline expression chains or pure dispatch shells).

Top-50-by-traffic coverage: 39/50. Missing 11 are either trivial leaf
helpers with no useful locals (`CosFloat12bit`, `SinFloat12bit`) or have
all locals inside `switch` case scopes that we deliberately skipped to
avoid noise (`Screen_RaceResults`).

Wave 2 consumption: for each row, the Ghidra decomp at `orig_address`
will show a `local_8` / `iVar3` / `puVar5` style auto-name where the
suggested name applies. Use `mcp__ghidra__variable_rename` with the orig
addr and the Ghidra-side auto-name (which Wave 2 must resolve via
`mcp__ghidra__function_variables` first). Do NOT blindly apply — the
port's local may be a result of a different transformation; visually
confirm the role matches before pushing.
