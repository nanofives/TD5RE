# SAR vs SHR semantic audit — TD5_d3d.exe vs port

**Date:** 2026-05-20
**Result:** CLEAN PASS. No SAR-vs-SHR semantic divergences found in port.

## Methodology

1. Enumerate every `SAR` instruction in `TD5_d3d.exe` (Ghidra `search_instructions`).
2. Enumerate every `SHR` instruction (sanity check on usage pattern).
3. Audit port source for unsigned right-shift patterns that could break SAR semantics:
   - `(uint*)x >> N` casts with the cast applied **before** the shift.
   - Unsigned variable declarations initialized from `>> N` where the RHS could be signed.

## Orig binary findings

- **SAR**: ≥500 occurrences (search limit hit). Pervasive throughout physics/AI/cascade address range (0x401000–0x40A000). Shift counts dominated by `0x8` (24.8 fixed-point → integer), `0xC` (12-bit angle units), with `0x1`, `0x2`, `0x4`, `0x5`, `0xA`, `0xB` also common.
- **SHR**: ≥500 occurrences. **Critical observation:** in the physics address range (0x401000–0x40A000), every SHR is `SHR reg, 0x1F` — the MSVC sign-bit extraction idiom emitted as part of the truncate-toward-zero pattern:
  ```
  CDQ                     ; or
  SAR EAX, 31             ; signed_int >> 31 = -1 if negative, 0 if positive
  ; or for unsigned-then-mask:
  MOV EBX, EAX
  SHR EBX, 0x1F           ; bit-31 isolated as 0 or 1
  AND EBX, mask           ; pre-bias for round-toward-zero
  ADD EAX, EBX
  SAR EAX, shift
  ```
  Non-`>> 31` SHR sites are concentrated in render/audio/asset code (>0x411000) — color byte extraction, audio mixing, decompression — where logical shift is correct because the operand is genuinely unsigned data.

**Implication:** every meaningful right-shift of physics/game state in the orig is SAR (signed). The port must preserve that.

## Port-side audit

Searched `td5mod/src/td5re/*.c` for the patterns:

```
^\s*(uint\w+|unsigned)\s+\w+\s*=.*>>\s*[1-9]
\((uint32_t|uint16_t|unsigned[^_a-zA-Z])\)\s*[a-zA-Z_]\w*\s*>>\s*[1-9]
```

### Proof for the vfx lateral_slip case

Claim: `(uint8_t)((uint16_t)x >> 2)` equals `(uint8_t)(x >> 2)` for any `int32_t x`, where the latter uses SAR.

Let `b16 = (uint16_t)x` (bit pattern of low 16 bits of x, sign-extended to int and reinterpreted unsigned). Then:
- Port: `(b16 >> 2)` — logical shift; result in `[0, 0x3FFF]`. Cast to `uint8_t` = low 8 bits.
- Orig: `x >> 2` (SAR) = `floor(x / 4)`. Cast to `uint8_t` = `floor(x/4) mod 256`.

Difference: `(b16 >> 2) - floor(x/4)` is a multiple of `2^14 = 16384`, which is a multiple of 256. Therefore `low_8_bits(port) == low_8_bits(orig)`. ✓

### Port's intentional SAR-emulating macros

`td5_physics.c:2734-2736` defines:
```c
#define SAR12(x) (((x) + (((x) >> 31) & 0xFFF)) >> 12)
#define SAR10(x) (((x) + (((x) >> 31) & 0x3FF)) >> 10)
#define SAR8_U8(x) (((x) + (((x) >> 31) & 0xFF)) >> 8)
```

This is the MSVC compiler's exact codegen pattern for `signed_int / N` where N is a power of two, producing **truncate-toward-zero** semantics (matching ANSI C integer division). Confirmed by the function header comment: "every SAR uses truncate-toward-zero rounding".

The orig binary's SAR sites are paired with `+ ((x>>31) & mask)` pre-bias in MSVC-emitted code — so the orig itself produces truncate-toward-zero from SAR+bias. The port mirrors this exactly. **Byte-faithful.**

## Conclusion

**The SAR-vs-SHR axis is RULED OUT as a source of port-vs-orig divergence.** The port has been disciplined about preserving signed-shift semantics, and where unsigned casts appear, they are either:
- on genuinely unsigned data (color, byte extraction), or
- followed by mask/truncate operations that make the SAR/SHR difference invisible.

This narrows the search space for remaining convergence residue. Combined with the FPU CW finding ([[reference_fpu_control_word_arch_divergence_2026-05-20]]), two of the three instruction-semantics axes are now characterized:

| Axis | Status |
|---|---|
| FPU control word (PC + RC) | Diverges — fix ready: `_controlfp_s(_PC_64 \| _RC_DOWN, ...)` |
| Signed right shift (SAR vs SHR) | **CLEAN** — no port bugs |
| FISTP/`__ftol` float-to-int | Subsumed by FPU CW finding (closes after RC=RDN fix) |

## Remaining instruction-semantics audits

Lower priority but still worth running once the FPU CW fix lands:

1. **IDIV / signed division by non-power-of-two.** Likely already correct under GCC, but worth a spot-check on any `int a / int b` where b is variable.
2. **IMUL overflow.** If port compiled without `-fwrapv`, signed overflow is UB; GCC may have optimized branches that depend on overflow wrap. Run `grep -rn '\* [a-zA-Z]' td5mod/src/td5re/td5_physics.c | head -20` to inventory multiply sites.
3. **CDQ/CWD sign-extension.** Paired with IDIV; covered by #1.
4. **ROL/ROR.** Search Ghidra for `ROL`/`ROR` mnemonics; check port doesn't translate as `(x << N) | (x >> (32-N))` without ensuring N is non-zero (undefined when N=0 for shift counts equal to width).

If the FPU CW fix lands and convergence residue persists, IMUL/IDIV signed-overflow audit is the next stop.
